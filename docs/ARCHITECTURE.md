# Architecture

## 1. Overall Goal

This workspace combines two layers:

- the upstream `elfin_robot_ros2` package set
- a lightweight visualization/control overlay in the workspace root

The overlay gives you a simpler entrypoint for RViz-based model display and joint-angle testing, while keeping the upstream Elfin packages buildable with `colcon build`.

## 2. Main Packages And Scripts

### Workspace Root

- `run_elfin_visualize.sh`
  Entry script. Cleans stale processes, sources ROS2 Humble and the local workspace, then launches visualization.
- `elfin_visualize.launch.py`
  Runtime launch file. Resolves the selected robot model, generates a temporary URDF and RViz config, and starts the key nodes.
- `elfin_visualize.rviz`
  RViz template used to generate the per-model runtime RViz config.
- `control_elfin.py`
  Tk control panel. Lets the user move joint sliders, publish `joint_states`, and relaunch with a different robot model.

### Core Source Tree

- `src/elfin_robot_ros2/elfin_description`
  Robot descriptions, meshes, Xacro files, and display launch files.
- `src/elfin_robot_ros2/elfin*_ros2_gazebo`
  Per-model Gazebo simulation packages.
- `src/elfin_robot_ros2/elfin*_ros2_moveit2`
  Per-model MoveIt2 packages.
- `src/elfin_robot_ros2/elfin_basic_api`
  High-level robot control API and GUI-related logic.
- `src/elfin_robot_ros2/elfin_ethercat_driver`
  EtherCAT driver for real hardware.
- `src/elfin_robot_ros2/elfin_ros_control`
  ros2_control hardware interfaces.
- `src/elfin_robot_ros2/elfin_robot_bringup`
  Hardware bringup configs and launch files.
- `src/elfin_robot_ros2/elfin_robot_msgs`
  Custom services/messages.
- `src/elfin_robot_ros2/soem_ros2`
  SOEM dependency packaged in-tree.

## 3. Runtime Nodes In The Visualizer Flow

When you run:

```bash
bash run_elfin_visualize.sh elfin5
```

the main runtime path is:

1. `run_elfin_visualize.sh`
2. `ros2 launch elfin_visualize.launch.py model:=elfin5`
3. `robot_state_publisher`
4. `control_elfin.py`
5. `rviz2`

### Node Roles

- `robot_state_publisher`
  Consumes `robot_description` and `joint_states`, publishes TF and TF static.
- `elfin_controller_panel` in `control_elfin.py`
  Subscribes to `/joint_states`, keeps the latest pose, and publishes commanded `joint_states` in visualization mode.
- `rviz2`
  Visualizes the robot model and TF tree.

## 4. Information Flow

### Visualization Mode

The information flow is:

1. `elfin_visualize.launch.py` selects model-specific Xacro
2. Xacro is expanded into a temporary URDF in `/tmp`
3. The same URDF is passed to `robot_state_publisher`
4. The RViz template is rendered into a temporary config that points to that URDF file
5. `control_elfin.py` publishes `/joint_states`
6. `robot_state_publisher` transforms joint states into `/tf` and `/tf_static`
7. `rviz2` renders the robot from URDF + TF

### Why This Is More Stable

Previously, RViz could depend on `/robot_description` topic timing and GUI-thread-driven state publishing. The current design avoids both fragile points:

- RViz reads a concrete URDF file instead of waiting on topic timing
- `joint_states` publishing runs in a background thread instead of inside Tk UI timing

## 5. Model Switching Flow

Changing the model from the control-panel dropdown does not just change slider limits.

It triggers a full relaunch:

1. `control_elfin.py` starts `run_elfin_visualize.sh <new_model>`
2. the current control panel closes
3. old `rviz2`, `robot_state_publisher`, and joint-state publishers are cleaned
4. a new URDF and RViz config are generated for the selected model
5. the new visualization chain starts cleanly

This is important because model switching affects:

- robot description
- RViz model source
- joint limits
- displayed geometry

## 6. Topics And Their Purpose

### Common Topics In Visualization Flow

- `/joint_states`
  Published by `control_elfin.py` in visualization mode; consumed by `robot_state_publisher`
- `/tf`
  Dynamic transforms generated by `robot_state_publisher`
- `/tf_static`
  Static transforms generated by `robot_state_publisher`

### In MoveIt / Hardware Flows

Depending on the selected launch path from upstream packages, the workspace also uses:

- `/elfin_arm_controller/follow_joint_trajectory`
  Action interface for trajectory execution
- MoveIt planning scene topics
- EtherCAT-related state and service interfaces

## 7. Practical Usage Paths

### Path A: Lightweight RViz Visualization

Use:

```bash
bash run_elfin_visualize.sh elfin5
```

Use this when you want:

- quick model visualization
- joint slider testing
- debugging model selection or RViz display behavior

### Path B: Gazebo + MoveIt2

Use the upstream per-model packages, for example:

```bash
ros2 launch elfin5_ros2_moveit2 elfin5.launch.py
```

Use this when you want:

- motion planning
- Gazebo simulation
- richer controller integration

### Path C: Real Hardware

Use `elfin_robot_bringup`, `elfin_ethercat_driver`, and `elfin_ros_control` paths.

Use this when you want:

- EtherCAT communication
- real robot bringup
- ros2_control hardware integration

## 8. What The Repository Can Do

At a repository level, this workspace supports:

- building all Elfin ROS2 packages with `colcon`
- RViz-based robot visualization
- model-specific robot descriptions
- MoveIt2 planning configuration for multiple Elfin variants
- Gazebo simulation packages
- real-hardware communication through EtherCAT
- high-level basic API / GUI control flows

## 9. Recommended Reading Order

If you are new to the repository:

1. read the root `README.md`
2. run `colcon build`
3. try `bash run_elfin_visualize.sh elfin5`
4. read `src/elfin_robot_ros2/README.md`
5. then inspect the specific package you care about: `elfin_description`, `elfin_basic_api`, `elfin_ros_control`, or `elfin_ethercat_driver`
